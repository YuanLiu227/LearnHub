# LearnHub 学习内容搜索平台 — 项目总结

## 项目背景

本项目最初定位为"AI 热点监控系统"（Hot Monitor），用于自动采集 AI 领域热点资讯。在开发过程中逐步演变为**多源学习内容搜索平台**（LearnHub），核心功能从"热点监控"转向"关键词驱动的学习资源搜索 + 博主内容自动收集"，并引入了完整的用户系统、收藏管理、归档/恢复、选择式批量删除等资源生命周期管理功能、独立的管理后台、用户可配置的设置系统以及机场订阅自动代理功能。

## 架构演进

### Phase 1 — 热点监控系统原型
- 定时抓取 Hacker News、Reddit、Google News 等 AI 热点
- 无用户系统，无持久化搜索
- 简单的前端表格展示

### Phase 2 — 学习资源搜索平台
- 引入关键词管理：添加、启用/暂停、归档/恢复
- 多源搜索：Bilibili、YouTube、编程导航、鱼皮AI导航
- 综合评分 + Bigram Jaccard 去重
- 异步搜索 + 轮询进度
- 质量阈值过滤（播放量/点赞/收藏）

### Phase 3 — 博主关注系统
- 关注 Bilibili UP 主 / YouTube 频道
- 博主内容收集独立于关键词搜索，不设质量阈值
- Bilibili 空间 API + 搜索 API 双层回退
- 每日 08:00 自动收集（node-cron）

### Phase 4 — 资源管理增强
- 收藏/取消收藏 + 级联删除保护
- 完成标记（绿色边框 + 删除线）
- 选择式批量删除 + 一键清空
- 两级删除：归档（软删除）vs 永久删除（硬删除）
- 知识搜索（全文搜索已有资源）

### Phase 5 — 多用户系统
- 邮箱注册两步验证（图形验证码 → 邮箱验证码）
- SVG 图形验证码（4 位数字，一次性使用，5 分钟过期）
- 邮箱验证码（6 位数字，10 分钟过期，SMTP 发送）
- JWT 认证（7 天过期），用户数据 `AND user_id = ?` 完全隔离
- 密码强度校验（字母、数字、特殊字符至少 2 类）

### Phase 6 — MCP 集成
- Context7 MCP：获取最新技术文档和代码示例

### Phase 7 — 视频 URL 提交
- 直接粘贴 Bilibili/YouTube 链接，自动提取视频信息
- 复用已有的 `extractBVID()` 和视频统计函数
- 新增 `resource_type` 列（keyword / creator / direct_video）
- 资源总览新增「视频资源」子标签页
- 批量清除功能扩展支持 `direct_video` 类型

### Phase 8 — 管理后台系统
- 独立 HTML 管理页面，由 Express 在 `/admin` 路由静态提供
- users 表新增 `role`（admin/user）和 `status`（active/frozen）列
- JWT 签发加入 `role` 字段，adminRequired 中间件
- 管理员自动分配（邮箱匹配 SMTP_USER → role = admin）
- 用户管理：分页列表、冻结/解冻、删除
- 系统设置：注册开关

### Phase 9 — 用户配置系统
- 每个用户独立配置 API 密钥（DeepSeek、YouTube）
- 四个数据源开关（Bilibili、YouTube、编程导航、鱼皮AI导航）
- `user_config` 表：复合主键 `(key, user_id)`
- AES-256-GCM 加密存储 API 密钥
- 三层回退：用户配置 → 环境变量 → 默认值
- 来源追踪：'user' | 'env' | 'none' | 'default'
- 密钥响应自动掩码

### Phase 10 — 代理管理系统
- 新增 `YOUTUBE_SUBSCRIPTION_URL` 用户配置项
- 用户输入机场订阅链接后，服务器自动启动 mihomo 实例
- proxy-manager.ts 单例服务：管理 mihomo 进程生命周期
- 端口自动分配（15733~15833），同一订阅 URL 共享实例
- 服务器重启自动恢复所有代理实例
- mihomo 配置简化（去掉 GEOIP 和 url-test）
- 前端设置页移除手动代理输入框，改为订阅链接输入框

## 数据流

### 关键词搜索流
```
用户添加关键词 → 触发搜索
    ↓
Bilibili API（搜索 + 统计，含阈值过滤）
YouTube API（搜索 + 统计，含阈值过滤，通过 mihomo 代理）
编程导航（文章列表抓取）
鱼皮AI导航（课程列表抓取）
    ↓
综合评分 → Bigram Jaccard 去重
    ↓
存入 SQLite（resource_type='keyword', keyword_id 关联）
    ↓
前端展示（按热度排序）
```

### 博主内容收集流
```
用户关注博主 → 触发收集 / 每日 08:00 自动
    ↓
Bilibili：空间 API → 搜索 API 回退
YouTube：Search API + Video Stats（通过 mihomo 代理）
    ↓
时间过滤（近 7 天），不设质量阈值
    ↓
存入 SQLite（resource_type='creator', creator_id 关联）
    ↓
前端展示
```

### 视频 URL 提交流
```
用户选择平台 + 粘贴 URL → 提交
    ↓
提取视频 ID（BVID / YouTube Video ID）
    ↓
调用平台 API 获取视频信息
    ↓
去重检查 → 构造 news_item
（resource_type='direct_video'）
    ↓
存入 SQLite → 刷新资源列表
```

### 用户注册流
```
Step 1: 邮箱 + 密码 + 图形验证码
    → 校验图形验证码 → 检查注册开关 → 校验密码强度
    → 发送邮箱验证码（SMTP / devCode）
Step 2: 邮箱 + 邮箱验证码
    → 校验验证码 → bcrypt 哈希 → 创建用户
    → 邮箱匹配 SMTP_USER → role = admin
    → 返回 JWT（含 role）
```

### 用户配置读写流
```
GET /api/config/user
    → getEffectiveConfig(userId) 遍历 USER_CONFIG_KEYS
    → user_config 表有值 → source='user'（加密自动解密）
    → process.env 有值 → source='env'
    → ENABLE_* 键 → source='default'（值为 true）
    → 无值 → source='none'（值为 null）
    → SECRET_KEYS 自动掩码 → 返回前端

PUT /api/config/user { key, value }
    → SECRET_KEYS → 加密存储
    → YOUTUBE_SUBSCRIPTION_URL → 同时触发 proxyManager.register()
    → value=null → 删除配置项
```

### 代理管理流
```
用户输入机场订阅链接
    ↓
PUT /api/config/user { key: "YOUTUBE_SUBSCRIPTION_URL", value: "https://..." }
    ↓
AES-256-GCM 加密 → 存入 user_config 表
    ↓
proxyManager.register(userId, url)
    ↓
MD5(url)[:8] hash → 检查是否已存在
    ↓
已存在 → 加入共享实例
不存在 → 分配端口 15733+ → 生成 mihomo config.yaml
    ↓
spawn('mihomo', ['-d', configDir])
    ↓
检测输出 "mixed proxy" / "started" → 确认启动（3 秒 fallback）
    ↓
自动写入 YOUTUBE_PROXY_URL = http://127.0.0.1:{port}
    ↓
YouTube API 调用 → getProxyAgent(userId) → mihomo 代理
```

### 管理后台认证流
```
访问 /admin → 输入邮箱+密码+图形验证码 → POST /auth/login
    ↓
JWT 含 role → 校验 role === 'admin'
    ↓
管理员 → 用户管理面板（冻结/解冻/删除/注册开关）
非 admin → 提示"该账号不是管理员"
```

### 收藏/取消收藏流
```
收藏 → PATCH resource/:id { favorited: true }
    → 从总览移除 → 仅在收藏标签页显示

取消收藏 → PATCH resource/:id { favorited: false }
    → 检查 keyword_id / creator_id 对应实体是否存在
    → 全部不存在 → 级联删除该资源
    → 任一存在 → 回到资源总览
```

## 关键设计决策

### 1. 博主收集与关键词搜索分离
博主收集不经过评分系统，不设质量阈值。两种来源在资源列表统一展示，通过 `keyword_id` / `creator_id` 区分。

### 2. 博主内容仅按时间筛选
关注博主的目的是获取最新内容而非最热内容。Bilibili 和 YouTube 均只取近 7 天视频，不设播放量/点赞阈值。

### 3. SVG 图形验证码
服务端内存存储，4 位数字 + 随机颜色 + 干扰线 + 干扰点 + 旋转。一次性使用，5 分钟 TTL，前端 SVG base64 data URI 渲染。

### 4. 邮箱两步验证
- Step 1：校验图形验证码 + 注册开关 + 邮箱 + 密码强度 → 发送邮箱验证码
- Step 2：校验验证码 + 注册开关 → 创建用户
- 开发模式（SMTP 未配置）自动输出 devCode

### 5. 数据隔离
所有表含 `user_id`，所有查询强制 `AND user_id = ?`，JWT 中间件自动注入。

### 6. 收藏 + 级联删除保护
取消收藏时检测关联实体是否存在。存在 → 回到总览；已删除 → 级联删除资源。

### 7. 两层删除机制
归档（软删除，保留关联资源）vs 永久删除（硬删除，级联清理）。

### 8. 异步搜索 + 轮询进度
关键词搜索和博主收集均异步执行，前端 500ms 轮询进度。任务完成后 5 分钟自动清理。

### 9. Bilibili API 多层回退
空间 API 限流时自动回退到搜索 API，按发布排序过滤该 UP 主视频。

### 10. 数据库渐进式迁移
ALTER TABLE ADD COLUMN + try/catch，兼容新旧数据库。

### 11. 使用 node:sqlite
Node.js v24 内置 `node:sqlite`（DatabaseSync），替代 better-sqlite3 避免 node-gyp 编译问题。

### 12. 显式 resource_type 列
`'keyword'` / `'creator'` / `'direct_video'` 三种资源类型，渐进式迁移添加。

### 13. 视频 URL 提交复用现有 API
复用 `extractBVID()` / `getBilibiliVideoStats()` / `getYouTubeVideoStats()`。

### 14. 管理后台与主前端分离
管理后台是独立 HTML（非 SPA），由 Express 静态服务在 `/admin` 路由提供。localStorage 独立管理 JWT token。

### 15. 管理员自动分配
注册邮箱与 SMTP_USER 匹配 → 自动 role = admin，无需新增独立的环境变量。

### 16. JWT 含角色信息
Payload：`{ userId, email, role }`，adminRequired 中间件先 authRequired 再校验 role。

### 17. 用户配置三层回退
user_config 表 → process.env → 默认值。来源追踪通过 `{ value, source }` 对象返回前端。

### 18. AES-256-GCM 加密存储
随机 IV + AuthTag + 密文，加密密钥通过 CONFIG_ENCRYPTION_KEY 派生（SHA-256）。格式：`hexIV:hexAuthTag:hexCiphertext`。

### 19. API 密钥响应掩码
- API 密钥：前 3 + `****` + 后 4
- 代理 URL：`http://***:***@hostname:port`
- 订阅链接：前 3 + `****` + 后 4

### 20. 用户自配置订阅 + 服务器动态代理
用户输入机场订阅链接 → 服务器自动启动 mihomo 实例：
- 同一订阅 URL 共享同一 mihomo 实例
- mihomo 简化配置（type: select + proxy-providers），避免 GEOIP 和 url-test
- 进程崩溃自动重启（2 秒延迟）
- 端口自动分配 15733~15833

## 计分公式（关键词搜索）

```
综合分 = 热度(heat) × 0.4 + 来源可信度(credibility) × 0.3
       + 时效性(timeliness) × 0.2 + 内容类型(contentType) × 0.1
```

| 维度 | 权重 | 规则 |
|------|------|------|
| 热度 | 0.4 | 播放量/点赞数归一化 0-100 |
| 来源可信度 | 0.3 | 编程导航/鱼皮AI=0.9, YouTube=0.8, Bilibili=0.6 |
| 时效性 | 0.2 | 7天=1.0, 30天=0.7, 90天=0.4, 更早=0.2 |
| 内容类型 | 0.1 | AI课程=0.9, 教程=0.8, 视频=0.7 |

- 后端：localhost:3001（Express API）
- 管理后台：http://localhost:3001/admin
- Vite proxy：/api → localhost:3001

## 部署要点

1. 必需：`JWT_SECRET`、`CONFIG_ENCRYPTION_KEY`（生产环境务必自定义）
2. 搜索需用户在前端自行配置 API 密钥
3. YouTube 需要 API Key + 机场订阅链接（国内环境）
4. 管理员由 SMTP_USER 匹配邮箱自动分配
5. SQLite 不适合 Serverless，推荐长时运行平台
6. 定时任务（博主收集）依赖 node-cron，需保持进程运行
7. 前端构建产物 client/dist/，由 Express serve 或 Nginx 部署
8. 服务器需安装 mihomo（Clash Meta 内核）支持代理功能
