# Hot Monitor 学习内容搜索平台 — 项目总结

## 项目背景

本项目最初定位为"AI 热点监控系统"（Hot Monitor），用于自动采集 AI 领域热点资讯。在开发过程中，逐步演变为**多源学习内容搜索平台**，核心功能从"热点监控"转向"关键词驱动的学习资源搜索 + 博主内容自动收集"，并引入了收藏管理、归档/恢复、选择式批量删除等完整的资源生命周期管理。

## 数据流

### 关键词搜索流
```
用户添加关键词 → 启用关键词 → 触发搜索
                    ↓
          ┌─── Bilibili API（搜索 + 统计数据）
          ├─── YouTube API（多语言搜索 + 统计数据，含阈值过滤）
          ├─── 编程导航 API
          └─── 鱼皮AI导航 API
                    ↓
              综合评分计算
                    ↓
              存入 SQLite
                    ↓
              前端展示关键词资源列表
```

### 博主内容收集流
```
用户关注博主 → 触发收集
                    ↓
          ┌─── Bilibili：空间 API → 搜索 API 回退
          └─── YouTube：Search API + Video Stats API
                    ↓
              时间过滤（近 7 天）
              不经过热度/质量阈值
                    ↓
              存入 SQLite
                    ↓
              前端展示在博主资源标签页
```

### 收藏/取消收藏流
```
用户收藏资源
    ↓
从资源总览移除 → 仅出现在收藏标签页

用户取消收藏
    ↓
检查关联实体存在性
    ├── 实体存在 → 回到资源总览
    └── 实体已删除 → 级联删除该资源
```

### 批量删除流
```
用户点击「选择」→ 进入选择模式
                    ↓
用户勾选目标资源 → 底部浮动栏显示已选数量
                    ↓
点击「批量删除」→ 确认
                    ↓
POST /api/dashboard/resources/batch-delete-by-ids
                    ↓
              刷新资源列表

用户点击「清空」→ 确认
                    ↓
POST /api/dashboard/resources/batch-delete
                    ↓
              刷新资源列表
```

## 关键设计决策

### 1. 博主收集与关键词搜索分离
博主内容收集**独立于关键词搜索**，不经过评分系统，不使用热度/质量阈值。两种数据源在 UI 上以子标签页区分（「关键词资源」/「博主资源」），但在统一列表中展示。

### 2. 博主内容仅按时间筛选
关注博主的意义在于获取**最新**内容而非最热内容。Bilibili 和 YouTube 的博主收集逻辑一致：只取近 7 天的最新视频，不设播放量/点赞阈值。

### 3. 收藏 + 级联删除保护
收藏提供"暂存"语义：收藏的资源从主列表隐藏，避免干扰。取消收藏时自动检测关联实体是否仍存在：
- 实体存在 → 回到主列表
- 实体已永久删除 → 资源级联删除

### 4. 两层删除机制（归档 vs 永久删除）
- **归档**：软删除，保留已关联的资源（适合暂时不需要的场景）
- **永久删除**：硬删除实体及其所有关联资源（适合废弃场景）
- 归档的资源/博主可在「总览」标签页一键恢复

### 5. 选择式批量删除 + 一键清空
- **选择模式**：进入选择模式后勾选多个资源，确认后只删除选中的资源
- **清空**：确认后删除当前标签页全部资源
- 两种功能独立，适应不同使用场景

### 6. AI 服务（DeepSeek 官方 API）
使用 DeepSeek 官方 API（DeepSeek V4 Flash 模型）进行内容摘要和质量评估。由 SiliconFlow 迁移至 DeepSeek 直连，减少中间层依赖。

### 7. 异步搜索 + 轮询进度
搜索和博主收集均异步执行，前端通过轮询 `/api/monitor/progress/:id` 或 `/api/creators/collect/progress/:id` 获取实时进度。

### 8. Bilibili API 多层回退
Bilibili 空间 API 存在限流，系统自动在空间 API 失败时回退到搜索 API，按发布排序过滤出该 UP 主的视频。

## 工程问题修复

| 问题 | 修复方式 |
|------|----------|
| 3 个死服务文件（firecrawl/twitter） | 删除 |
| `verifiedCount` 永不递增 | 在 DB insert 成功后自增 |
| 循环内动态 import | 改为文件顶部静态 import |
| Nodemailer 模块级创建 | 改为惰性初始化 |
| `as any` 滥用（30+ 处） | 替换为具体类型接口 |
| ProgressEmitter 内存泄漏 | 完成/错误后定时清理 |
| 数据库路径硬编码 | 支持 `DB_PATH` 环境变量覆盖 |
| 前端 3 个死组件 | 删除 |
| 接口重复声明 | 删除冗余声明 |
| 文件名误导 | `openrouter.ts` → `siliconflow.ts` → `deepseek.ts` |
| 取消收藏实体检查逻辑错误 | `!item.keyword_id` 在 null 时始终为 true，改为条件分支判断 |
| 收藏资源在删除关联实体后仍显示在总览 | fetchResources 过滤 `favorited` 项 |
| YouTube 博主内容被点赞阈值过滤 | 移除博主收集路径中的 view/like 阈值 |

## 开发环境

- Node.js >= 18
- npm workspaces（client/ + server/）
- 前端端口：5173
- 后端端口：3001
- Vite proxy 将 `/api` 转发到后端

## 部署要点

1. 必需环境变量：`DEEPSEEK_API_KEY`
2. YouTube 功能需要：`YOUTUBE_API_KEY` + `YOUTUBE_PROXY_URL`（国内）
3. 数据库文件默认生成在 `server/data/hot-monitor.db`
4. 可通过 `DB_PATH` 环境变量自定义数据库路径
5. 前端构建产物在 `client/dist/`，可部署到任意静态服务器
6. 由于使用 SQLite，不适合 Vercel Serverless 部署，推荐 Railway / Render 等长时运行平台
