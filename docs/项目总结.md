# LearnHub 学习内容搜索平台 — 项目总结

## 项目背景

本项目最初定位为"AI 热点监控系统"（Hot Monitor），用于自动采集 AI 领域热点资讯。在开发过程中，逐步演变为**多源学习内容搜索平台**（LearnHub），核心功能从"热点监控"转向"关键词驱动的学习资源搜索 + 博主内容自动收集"，并引入了完整的用户系统、收藏管理、归档/恢复、选择式批量删除等资源生命周期管理功能、独立的管理后台以及用户可配置的设置系统。

## 架构演进

### Phase 1 — 热点监控系统（Hot Monitor 原型）
- 定时抓取 Hacker News、Reddit、Google News 等 AI 热点
- 无用户系统，无持久化搜索
- 简单的前端表格展示

### Phase 2 — 学习资源搜索平台（LearnHub 转型）
- 引入关键词管理：添加、启用/暂停关键词
- 多源搜索：Bilibili、YouTube、编程导航、鱼皮AI导航
- 综合评分 + 去重算法
- 异步搜索 + 轮询进度

### Phase 3 — 博主关注系统
- 关注 Bilibili UP 主 / YouTube 频道
- 博主内容收集独立于关键词搜索
- Bilibili 多层 API 回退策略
- 定时任务（每天早上 08:00 自动收集）

### Phase 4 — 资源管理增强
- 收藏/取消收藏 + 级联删除保护
- 完成标记
- 选择式批量删除 + 一键清空
- 两级删除（归档 vs 永久删除）
- 知识搜索（全文搜索已有资源）

### Phase 5 — 多用户系统
- 邮箱注册（两步验证：图形验证码 → 邮箱验证码）
- SVG 图形验证码（4 位数字，一次性使用，5 分钟过期）
- 邮箱验证码（6 位数字，10 分钟过期，SMTP 发送）
- JWT 认证（7 天过期），用户数据完全隔离
- 密码强度校验（字母、数字、特殊字符至少两种）

### Phase 6 — MCP 集成
- Context7 MCP：获取最新技术文档

### Phase 7 — 视频 URL 提交
- 直接粘贴 Bilibili/YouTube 视频链接，自动提取视频信息
- 复用现有提取 BVID 和 YouTube Video ID 的函数
- 新增 `resource_type` 列区分三种资源来源（keyword / creator / direct_video）
- 资源总览新增「视频资源」子标签页
- 「关键词」页面新增「添加视频」子标签，含平台选择和 URL 输入
- 批量清除功能扩展支持 `type='direct_video'`
- 知识搜索自动覆盖视频资源（基于统一的 news_items 表查询）

### Phase 8 — 管理后台系统
- 独立的管理后台 HTML 页面，由 Express 在 `/admin` 路由下静态提供，与主前端 SPA 完全分离
- users 表新增 `role`（admin/user）和 `status`（active/frozen）列
- JWT 签发加入 `role` 字段，adminRequired 中间件保护管理 API
- 管理员自动分配：注册邮箱与 SMTP_USER 匹配时自动设为 admin 角色
- 用户管理：分页列表、冻结/解冻、删除（含防自删、防删最后管理员、级联清理）
- 系统设置：注册开关，关闭后所有注册入口返回友好提示
- 冻结账号检查：frozen 状态的用户拒绝登录
- 管理后台使用 localStorage 独立管理 JWT token，与主前端互不影响
- 所有管理后台 API 使用 `adminRequired` 中间件保护（先 authRequired 再校验 role）

### Phase 9 — 用户可配置设置系统
- 每个用户独立配置 API 密钥（DeepSeek、YouTube）、代理地址（YouTube Proxy）
- 四个数据源开关（Bilibili、YouTube、编程导航、鱼皮AI导航）
- `user_config` 表：复合主键 `(key, user_id)`，独立于系统配置表
- AES-256-GCM 加密存储 API 密钥，解密后使用
- API 响应中自动掩码处理密钥字段
- `getUserConfigValue(key, userId?)` 模式：查 DB → 回退 env → 默认值
- 配置来源追踪：`'user' | 'env' | 'none' | 'default'`
- 前端设置页面：密码输入框 + 显示/隐藏切换 + 来源指示器 + 独立保存/清除按钮

## 数据流

### 关键词搜索流
```
用户添加关键词 → 启用关键词 → 触发搜索
                    ↓
          ┌─── Bilibili API（搜索 + 统计数据，含阈值过滤）
          ├─── YouTube API（多语言搜索 + 统计数据，含阈值过滤）
          ├─── 编程导航 API（文章列表抓取）
          └─── 鱼皮AI导航 API（课程列表抓取）
                    ↓
              综合评分计算 → 去重（Bigram Jaccard）
                    ↓
              存入 SQLite（标记 user_id）
                    ↓
              前端展示资源列表（按热度排序）
```

### 博主内容收集流
```
用户关注博主 → 触发收集（或每日 08:00 自动）
                    ↓
          ┌─── Bilibili：空间 API → 搜索 API 回退
          └─── YouTube：Search API + Video Stats API
                    ↓
              时间过滤（近 7 天）
              不经过热度/质量阈值
                    ↓
              存入 SQLite（标记 creator_id）
                    ↓
              前端展示博主资源
```

### 用户注册流
```
用户输入邮箱 + 密码 + 图形验证码
                    ↓
          POST /auth/register
                    ↓
          校验图形验证码 → 检查注册开关 → 校验密码强度 → 检查邮箱是否已注册
                    ↓
          生成 6 位邮箱验证码 → 存入 verification_codes 表
                    ↓
          SMTP 已配置 → 发送邮件
          SMTP 未配置 → 返回 devCode（开发模式，控制台可查看）
                    ↓
          用户输入邮箱验证码（手动输入，不自动填入）
                    ↓
          POST /auth/register/verify
                    ↓
          检查注册开关 → 校验邮箱验证码 → bcrypt 哈希密码
                    ↓
          邮箱匹配 SMTP_USER → 自动设为 admin 角色
          邮箱不匹配 SMTP_USER → 设为 user 角色
                    ↓
          返回 JWT（含 role）→ 前端存储 token → 自动登录
```

### 视频 URL 提交流
```
用户选择平台（Bilibili / YouTube）+ 粘贴视频 URL
                    ↓
          POST /api/videos/submit
                    ↓
          提取视频 ID（BVID / YouTube Video ID）
                    ↓
          调用对应平台 API 获取视频统计信息
          ├── Bilibili：getBilibiliVideoStats(avid)
          └── YouTube：getYouTubeVideoStats(videoId, userId)
                    ↓
          检查 URL 是否已存在（去重）
                    ↓
          构造 news_item → resource_type = 'direct_video'
          keyword_id = null, creator_id = null
                    ↓
          存入 SQLite
                    ↓
          返回 { success, item } → 前端刷新资源列表
```

### 用户配置读写流
```
用户打开设置页面 → GET /api/config/user
                    ↓
          getEffectiveConfig(userId) 遍历 USER_CONFIG_KEYS
                    ↓
          查 user_config 表 → 有值 → 解密 → 标记 source='user'
          无值 → 查 process.env → 有值 → 标记 source='env'
          无值 → ENABLE_* 键 → 标记 source='default'（值为 true）
          均无 → 标记 source='none'（值为 null）
                    ↓
          SECRET_KEYS 字段自动掩码处理
                    ↓
          前端展示各配置项及来源指示
                    ↓
          用户修改 → PUT /api/config/user { key, value }
                    ↓
          SECRET_KEYS 自动加密存储 → 写入 user_config 表
          value=null → 删除该配置项 → 回退为系统默认
```

### 管理后台认证流
```
管理员访问 http://localhost:3001/admin
                    ↓
          输入邮箱 + 密码 + 图形验证码
                    ↓
          POST /api/auth/login → JWT 含 role
                    ↓
          检查 role === 'admin' → 进入管理面板
          非 admin → 前端拦截，提示"该账号不是管理员"
                    ↓
          管理面板展示用户列表（含分页）
          提供冻结/解冻/删除操作
          提供注册开关
```

### 收藏/取消收藏流
```
用户收藏资源
    ↓
从资源总览移除 → 仅在收藏标签页显示

用户取消收藏
    ↓
检查关联实体存在性
    ├── 实体存在 → 回到资源总览
    └── 实体已删除 → 级联删除该资源
```

## 关键设计决策

### 1. 博主收集与关键词搜索分离
博主内容收集**独立于关键词搜索**，不经过评分系统，不使用热度/质量阈值。两种数据源在资源列表中统一展示，通过 `keyword_id` 和 `creator_id` 区分来源。

### 2. 博主内容仅按时间筛选
关注博主的意义在于获取**最新**内容而非最热内容。Bilibili 和 YouTube 的博主收集逻辑一致：只取近 7 天的最新视频，不设播放量/点赞阈值。

### 3. SVG 图形验证码系统
- 服务端内存存储，不占用数据库
- 4 位数字 + 随机颜色 + 干扰线 + 干扰点 + 字符旋转
- 一次性使用：验证成功后立即从 Map 删除
- 定期清理：每分钟扫描过期验证码（5 分钟 TTL）
- 前端通过 SVG base64 data URI 渲染，点击图片刷新

### 4. 邮箱两步验证
注册流程拆分为两步，防止无效注册：
- **Step 1**：校验图形验证码 + 注册开关 + 邮箱格式 + 密码强度 → 发送邮箱验证码
- **Step 2**：校验注册开关 + 校验邮箱验证码 → 创建用户
- **开发模式**：SMTP 未配置时自动检测并返回 `devCode`，控制台可查看

### 5. 数据隔离架构
每个表都包含 `user_id` 字段，所有查询强制 `AND user_id = ?` 过滤：
- 不同用户的数据完全隔离
- 数据库层面保障安全性
- JWT payload 包含 `userId`，中间件自动注入

### 6. 收藏 + 级联删除保护
取消收藏时自动检测关联实体是否仍存在：
- 实体存在 → 回到主列表
- 实体已永久删除 → 资源级联删除

### 7. 两层删除机制（归档 vs 永久删除）
- **归档**：软删除，保留已关联的资源（适合暂时不需要的场景）
- **永久删除**：硬删除实体及其所有关联资源（适合废弃场景）
- 归档的资源/博主可在「总览」标签页一键恢复

### 8. 异步搜索 + 轮询进度
搜索和博主收集均异步执行，前端通过 500ms 间隔轮询获取实时进度。任务完成后 5 分钟自动清理进度数据，防止内存泄漏。

### 9. Bilibili API 多层回退
Bilibili 空间 API 存在限流，系统自动在空间 API 失败时回退到搜索 API，按发布排序过滤出该 UP 主的视频。

### 10. 数据库渐进式迁移
使用 `ALTER TABLE ADD COLUMN` 配合 `try/catch` 实现安全迁移，兼容新旧数据库文件。

### 11. 使用 node:sqlite 替代 better-sqlite3
Node.js v24 内置了 `node:sqlite` 模块，免去了 better-sqlite3 的 node-gyp 原生编译问题。API 兼容（`all()`、`get()`、`run()`、`exec()`），无需额外依赖。

### 12. 显式 resource_type 列
通过显式 `resource_type` 列（`'keyword'` / `'creator'` / `'direct_video'`），通过渐进式迁移添加到现有数据库。资源列表接口通过该字段分三类展示。

### 13. 视频 URL 提交复用现有数据源能力
视频 URL 提交功能复用已有的 `extractBVID()` / `getBilibiliVideoStats()` 和 `getYouTubeVideoStats()` 函数，共享同一套 API 适配器。热度计算共享归一化逻辑：`Math.min(100, Math.round(viewCount / 1000))`。

### 14. 管理后台与主前端完全分离
管理后台是一个独立的 HTML 页面（非 SPA），由 Express 静态文件服务在 `/admin` 路由下提供：
- **主前端**：`localhost:5173` — React SPA，面向普通用户
- **管理后台**：`localhost:3001/admin` — 独立 HTML/CSS/JS 页面，面向管理员
- 使用 localStorage 独立管理 JWT token，与主前端互不影响

### 15. 管理员自动分配
无需新增环境变量，直接复用 SMTP_USER 配置：注册时邮箱与 `SMTP_USER` 匹配 → 自动 `role = 'admin'`。

### 16. JWT 含角色信息
JWT payload 扩展为 `{ userId, email, role }`，`adminRequired` 中间件先验证 JWT 再校验 `role === 'admin'`。

### 17. 用户配置 Env 回退模式
用户配置服务采用三层回退策略：
- **用户层**：`user_config` 表（API 密钥加密存储）
- **环境变量层**：`process.env`（向后兼容）
- **默认层**：数据源开关默认 `true`
- 来源追踪通过 `{ value, source }` 对象返回，前端根据来源显示不同指示器

### 18. AES-256-GCM 加密存储 API 密钥
- 使用 Node.js 内置 `crypto` 模块，无需额外依赖
- 随机 IV + AuthTag 保证加密安全性
- 加密格式：`hexIV:hexAuthTag:hexCiphertext`
- `isEncrypted()` 函数通过正则 `^[0-9a-f]+:[0-9a-f]+:[0-9a-f]+$` 判断是否已加密
- 加密密钥通过 `CONFIG_ENCRYPTION_KEY` 环境变量配置，默认使用开发密钥

### 19. API 密钥响应掩码
- 在 `/api/config/user` 响应中，用户来源的密钥字段自动掩码
- API 密钥：显示前 3 位 + `****` + 后 4 位（如 `sk-****9244`）
- 代理 URL：仅掩码密码部分（`http://***:***@hostname:port`）

## 用户系统设计

### 角色体系

| 角色 | 权限 | 分配方式 |
|------|------|----------|
| `user` | 使用主前端所有功能 | 注册时默认分配 |
| `admin` | 主前端所有功能 + 管理后台 | 注册邮箱匹配 SMTP_USER 时自动分配 |

### 用户状态

| 状态 | 行为 |
|------|------|
| `active` | 正常使用 |
| `frozen` | 无法登录（登录时返回 403 "账号已被冻结，无法登录"） |

### 密码策略
- 最小长度：6 位
- 必须包含以下至少 2 类：字母（a-z, A-Z）、数字（0-9）、特殊字符
- 前后端双重校验
- bcrypt 哈希存储（10 轮 salt）

### JWT 认证
- 签发：登录/注册成功后签发
- Payload：`{ userId, email, role }`
- 过期时间：7 天
- 存储：localStorage，请求时通过 `Authorization: Bearer <token>` 携带
- 验证：`authRequired` 中间件自动校验，401 时前端自动跳转登录
- 角色验证：`adminRequired` 中间件在 `authRequired` 基础上校验 `role === 'admin'`

### 会话管理
- 页面刷新时从 localStorage 恢复 token
- 调用 `GET /api/auth/me` 验证 token 有效性
- token 无效/过期时自动清除并跳转登录页
- axios 拦截器自动注入 token 和处理 401 响应

## 综合评分公式（关键词搜索）

```
综合分 = 热度(heat) × 0.4
       + 来源可信度(credibility) × 0.3
       + 时效性(timeliness) × 0.2
       + 内容类型(contentType) × 0.1
```

| 维度 | 权重 | 取值规则 |
|------|------|----------|
| 热度 | 0.4 | 基于播放量/点赞数等指标归一化 |
| 来源可信度 | 0.3 | 编程导航/鱼皮AI导航=0.9, Bilibili=0.6, YouTube=0.8 |
| 时效性 | 0.2 | 7 天内=1.0, 30 天内=0.7, 90 天内=0.4, 更早=0.2 |
| 内容类型 | 0.1 | AI 课程=0.9, 教程文章=0.8, 视频=0.7 |

## 开发环境

- Node.js >= 20（推荐 v24 以使用内置 node:sqlite）
- npm workspaces（client/ + server/）
- 前端端口：5173（Vite dev server）
- 后端端口：3001（Express API）
- 管理后台地址：`http://localhost:3001/admin`
- Vite proxy 将 `/api` 转发到后端
- 开发模式验证码提示到控制台（不自动填入输入框）

## 部署要点

1. **必需环境变量**：`JWT_SECRET`
2. **生产环境**：务必设置 `CONFIG_ENCRYPTION_KEY` 为自定义密钥（默认密钥仅用于开发）
3. 搜索功能需要用户在设置页自行配置 API 密钥（DeepSeek、YouTube）及代理地址
4. 邮件功能需要配置 SMTP 环境变量（推荐 QQ 邮箱 SMTP），否则开发模式返回 `devCode`
5. **管理员账号由 SMTP_USER 决定**：配置 SMTP 后，使用 SMTP_USER 对应的邮箱注册自动成为管理员
6. 数据库文件默认在 `server/data/hot-monitor.db`，可通过 `DB_PATH` 自定义
7. 由于使用 SQLite，不适合 Vercel Serverless 部署，推荐 Railway / Zeabur 等长时运行平台
8. 前端构建产物在 `client/dist/`，Express 可直接 serve 或通过 Nginx 部署
9. 定时任务（博主内容收集）依赖于 node-cron，需保持进程长期运行
10. 管理后台是独立 HTML 文件，无需构建，直接由 Express 在 `/admin` 路由下静态提供
