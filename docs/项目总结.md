# LearnHub 学习内容搜索平台 — 项目总结

## 项目背景

本项目最初定位为"AI 热点监控系统"（Hot Monitor），用于自动采集 AI 领域热点资讯。在开发过程中，逐步演变为**多源学习内容搜索平台**（LearnHub），核心功能从"热点监控"转向"关键词驱动的学习资源搜索 + 博主内容自动收集"，并引入了完整的用户系统、收藏管理、归档/恢复、选择式批量删除等资源生命周期管理功能。

## 架构演进

### Phase 1 — 热点监控系统（Hot Monitor 原型）
- 定时抓取 Hacker News、Reddit、Google News 等 AI 热点
- 无用户系统，无持久化搜索
- 简单的前端表格展示

### Phase 2 — 学习资源搜索平台（LearnHub 转型）
- 引入关键词管理：添加、启用/暂停关键词
- 多源搜索：Bilibili、YouTube、编程导航、鱼皮AI导航
- 综合评分 + 去重算法
- 异步搜索 + 轮询进度

### Phase 3 — 博主关注系统
- 关注 Bilibili UP 主 / YouTube 频道
- 博主内容收集独立于关键词搜索
- Bilibili 多层 API 回退策略
- 定时任务（每天早上 08:00 自动收集）

### Phase 4 — 资源管理增强
- 收藏/取消收藏 + 级联删除保护
- 完成标记
- 选择式批量删除 + 一键清空
- 两级删除（归档 vs 永久删除）
- 知识搜索（全文搜索已有资源）

### Phase 5 — 多用户系统
- 邮箱注册（两步验证：图形验证码 → 邮箱验证码）
- SVG 图形验证码（4 位数字，一次性使用，5 分钟过期）
- 邮箱验证码（6 位数字，10 分钟过期，SMTP 发送）
- JWT 认证（7 天过期），用户数据完全隔离
- 密码强度校验（字母、数字、特殊字符至少两种）

## 数据流

### 关键词搜索流
```
用户添加关键词 → 启用关键词 → 触发搜索
                    ↓
          ┌─── Bilibili API（搜索 + 统计数据，含阈值过滤）
          ├─── YouTube API（多语言搜索 + 统计数据，含阈值过滤）
          ├─── 编程导航 API
          └─── 鱼皮AI导航 API
                    ↓
              综合评分计算 → 去重
                    ↓
              存入 SQLite（标记 user_id）
                    ↓
              前端展示资源列表
```

### 博主内容收集流
```
用户关注博主 → 触发收集（或每日 08:00 自动）
                    ↓
          ┌─── Bilibili：空间 API → 搜索 API 回退
          └─── YouTube：Search API + Video Stats API
                    ↓
              时间过滤（近 7 天）
              不经过热度/质量阈值
                    ↓
              存入 SQLite（标记 creator_id）
                    ↓
              前端展示博主资源
```

### 用户注册流
```
用户输入邮箱 + 密码 + 图形验证码
                    ↓
          POST /auth/register
                    ↓
          校验图形验证码 → 校验密码强度 → 检查邮箱是否已注册
                    ↓
          生成 6 位邮箱验证码 → 存入 verification_codes 表
                    ↓
          SMTP 已配置 → 发送邮件
          SMTP 未配置 → 返回 devCode（开发模式）
                    ↓
          用户输入邮箱验证码
                    ↓
          POST /auth/register/verify
                    ↓
          校验邮箱验证码 → bcrypt 哈希密码 → 创建用户
                    ↓
          返回 JWT → 前端存储 token → 自动登录
```

### 收藏/取消收藏流
```
用户收藏资源
    ↓
从资源总览移除 → 仅在收藏标签页显示

用户取消收藏
    ↓
检查关联实体存在性
    ├── 实体存在 → 回到资源总览
    └── 实体已删除 → 级联删除该资源
```

## 数据源对比

| 数据源 | 鉴权方式 | 内容特点 | 评分可信度 | 博主关注 |
|--------|----------|----------|-----------|---------|
| Bilibili | 无需密钥 | 视频教程，覆盖面广 | 0.6 | ✅（空间 API + 搜索回退） |
| YouTube | API Key | 国际视频教程，质量高 | 0.8 | ✅（Search API） |
| 编程导航 | 无需密钥 | 技术文章/教程，质量高 | 0.9 | ❌ |
| 鱼皮AI导航 | 无需密钥 | AI 课程/教程，质量高 | 0.9 | ❌ |

## 关键设计决策

### 1. 博主收集与关键词搜索分离
博主内容收集**独立于关键词搜索**，不经过评分系统，不使用热度/质量阈值。两种数据源在资源列表中统一展示，通过 `keyword_id` 和 `creator_id` 区分来源。

### 2. 博主内容仅按时间筛选
关注博主的意义在于获取**最新**内容而非最热内容。Bilibili 和 YouTube 的博主收集逻辑一致：只取近 7 天的最新视频，不设播放量/点赞阈值。

### 3. SVG 图形验证码系统
- 服务端内存存储，不占用数据库
- 4 位数字 + 随机颜色 + 干扰线 + 干扰点 + 字符旋转
- 一次性使用：验证成功后立即从 Map 删除
- 定期清理：每分钟扫描过期验证码（5 分钟 TTL）
- 前端通过 SVG base64 data URI 渲染，点击图片刷新

### 4. 邮箱两步验证
注册流程拆分为两步，防止无效注册：
- **Step 1**：校验图形验证码 + 邮箱格式 + 密码强度 → 发送邮箱验证码
- **Step 2**：校验邮箱验证码 → 创建用户
- **开发模式**：SMTP 未配置时自动检测并返回 `devCode`

### 5. 数据隔离架构
每个表都包含 `user_id` 字段，所有查询强制 `AND user_id = ?` 过滤：
- 不同用户的数据完全隔离
- 数据库层面保障安全性
- JWT payload 包含 `userId`，中间件自动注入

### 6. 收藏 + 级联删除保护
取消收藏时自动检测关联实体是否仍存在：
- 实体存在 → 回到主列表
- 实体已永久删除 → 资源级联删除

### 7. 两层删除机制（归档 vs 永久删除）
- **归档**：软删除，保留已关联的资源（适合暂时不需要的场景）
- **永久删除**：硬删除实体及其所有关联资源（适合废弃场景）
- 归档的资源/博主可在「总览」标签页一键恢复

### 8. 异步搜索 + 轮询进度
搜索和博主收集均异步执行，前端通过 500ms 间隔轮询获取实时进度。任务完成后 5 分钟自动清理进度数据，防止内存泄漏。

### 9. Bilibili API 多层回退
Bilibili 空间 API 存在限流，系统自动在空间 API 失败时回退到搜索 API，按发布排序过滤出该 UP 主的视频。

### 10. 数据库渐进式迁移
使用 `ALTER TABLE ADD COLUMN` 配合 `try/catch` 实现安全迁移，兼容新旧数据库文件。

## 用户系统设计

### 密码策略
- 最小长度：6 位
- 必须包含以下至少 2 类：字母（a-z, A-Z）、数字（0-9）、特殊字符
- 前后端双重校验
- bcrypt 哈希存储（10 轮 salt）

### JWT 认证
- 签发：登录/注册成功后签发
- Payload：`{ userId, email }`
- 过期时间：7 天
- 存储：localStorage，请求时通过 `Authorization: Bearer <token>` 携带
- 验证：`authRequired` 中间件自动校验，401 时前端自动跳转登录

### 会话管理
- 页面刷新时从 localStorage 恢复 token
- 调用 `GET /api/auth/me` 验证 token 有效性
- token 无效/过期时自动清除并跳转登录页
- axios 拦截器自动注入 token 和处理 401 响应

## 开发环境

- Node.js >= 20
- npm workspaces（client/ + server/）
- 前端端口：5173（Vite dev server）
- 后端端口：3001（Express API）
- Vite proxy 将 `/api` 转发到后端
- 开发模式验证码自动填入（SMTP 未配置时）

## 部署要点

1. **必需环境变量**：`DEEPSEEK_API_KEY`、`JWT_SECRET`
2. YouTube 功能需要：`YOUTUBE_API_KEY` + `YOUTUBE_PROXY_URL`（国内）
3. 邮件功能需要配置 SMTP 环境变量，否则开发模式始终返回 `devCode`
4. 数据库文件默认在 `server/data/hot-monitor.db`，可通过 `DB_PATH` 自定义
5. 由于使用 SQLite，不适合 Vercel Serverless 部署，推荐 Railway / Render 等长时运行平台
6. 前端构建产物在 `client/dist/`，Express 可直接 serve 或通过 Nginx 部署
7. 定时任务（博主内容收集）依赖于 node-cron，需保持进程长期运行
