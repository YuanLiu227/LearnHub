# LearnHub 学习内容搜索平台 — 需求文档

## 1. 项目概述

### 1.1 项目定位
LearnHub 是一个关键词驱动的多源学习资源搜索平台，支持博主内容自动收集和视频 URL 提交，帮助开发者从多个技术平台快速检索和发现高质量学习内容。提供独立的管理后台用于用户和系统管理。

### 1.2 目标用户
- 想要系统性学习某一技术领域的开发者
- 需要跨平台追踪特定博主/频道最新内容的用户
- 希望收藏和管理学习资源的用户
- 系统管理员（通过独立管理后台管理用户和系统设置）

### 1.3 核心价值
- **多源聚合**：一次搜索覆盖 Bilibili、YouTube、编程导航、鱼皮AI导航
- **博主追踪**：关注 UP 主/频道，自动收集最新内容（仅按时间筛选）
- **视频提交**：直接粘贴 Bilibili/YouTube 视频链接，自动提取信息并入库
- **资源管理**：收藏、完成标记、归档/恢复、选择式批量删除、一键清空
- **用户配置**：每个用户可独立配置自己的 API 密钥、代理地址和数据源开关
- **数据隔离**：多用户系统，每个用户独立管理自己的数据
- **管理后台**：独立的管理界面，支持账号管理、冻结/解封、注册控制

## 2. 用户系统

### 2.1 角色体系

| 角色 | 说明 | 分配方式 |
|------|------|----------|
| `user` | 普通用户 | 注册时默认分配 |
| `admin` | 管理员 | 注册邮箱与 `SMTP_USER` 匹配时自动分配 |

管理员通过访问 `http://localhost:3001/admin` 进入独立的管理后台。

### 2.2 用户状态

| 状态 | 说明 |
|------|------|
| `active` | 正常状态，可正常登录和使用 |
| `frozen` | 已冻结，无法登录系统 |

### 2.3 注册流程（两步验证）

```
Step 1: 提交邮箱 + 密码 + 图形验证码
                ↓
        校验图形验证码 → 检查注册开关 → 校验密码强度 → 检查邮箱是否已注册
                ↓
        生成 6 位邮箱验证码 → 存入 verification_codes 表
                ↓
        SMTP 已配置 → 发送邮件到注册邮箱
        SMTP 未配置 → 返回 devCode（开发模式）
                ↓
Step 2: 提交邮箱 + 邮箱验证码
                ↓
        检查注册开关 → 校验邮箱验证码 → bcrypt 哈希密码 → 创建用户
                ↓
        邮箱匹配 SMTP_USER → 自动设为 admin 角色
        邮箱不匹配 SMTP_USER → 设为 user 角色
                ↓
        返回 JWT（含 role）→ 前端存储 token → 自动登录
```

| 功能 | 描述 |
|------|------|
| 邮箱注册 | 使用邮箱作为账号，注册时发送验证码到邮箱验证 |
| 密码安全 | 密码需包含字母、数字、特殊字符中的至少两种，长度至少 6 位 |
| 图形验证码 | 4 位数字 SVG 验证码，防止机器人批量注册，一次性使用，5 分钟过期 |
| 邮箱验证码 | 6 位数字，10 分钟有效期，存储于数据库 verification_codes 表 |
| 注册开关 | 管理员可在后台关闭注册，关闭后所有注册入口返回"当前服务已停用" |
| JWT 认证 | 登录后签发 JWT（含 role），7 天过期，存储于 localStorage |

### 2.4 登录流程

| 功能 | 描述 |
|------|------|
| 邮箱+密码登录 | 需输入图形验证码，bcrypt 校验密码 |
| 冻结检查 | 登录时检查用户状态，frozen 状态返回 403 拒绝登录 |
| JWT 持久化 | Token 存储在 localStorage，通过 Authorization header 携带 |
| 自动登录恢复 | 页面刷新时读取 localStorage 的 token，调用 GET /auth/me 验证有效性 |
| 401 自动跳转 | 所有 API 响应 401 时自动清除 token 并跳转登录页 |

### 2.5 图形验证码系统

- 服务端内存存储（Map），key 为 captchaId
- 4 位数字，随机颜色、旋转、干扰线、干扰点
- SVG 格式输出，前端通过 `data:image/svg+xml;base64` 渲染
- **一次性使用**：验证成功后立即删除
- **自动清理**：每分钟扫描一次，清除过期（5 分钟）的验证码
- 登录、注册第一步、重新发送验证码均需图形验证码

### 2.6 邮箱验证码系统

- 存储于 `verification_codes` 表，记录邮箱、验证码、类型、过期时间、是否使用
- 10 分钟有效期，验证成功后标记 `used = 1`
- 通过 nodemailer + SMTP 发送邮件（支持 QQ 邮箱/163邮箱/Brevo/Resend 等）
- SMTP 未配置时自动返回 `devCode`，方便开发调试

## 3. 功能需求

### 3.1 关键词管理

| 功能 | 描述 |
|------|------|
| 添加关键词 | 用户输入关键词，系统存储并用于后续搜索 |
| 启用/暂停 | 切换关键词的监控状态，暂停的关键词不会被搜索 |
| 归档关键词 | 关键词归档后不再搜索，但已匹配的学习资源保留 |
| 恢复关键词 | 将已归档的关键词重新激活 |
| 永久删除 | 删除关键词及其关联的所有学习资源 |
| 关键词总览 | 展示所有关键词（含已归档），查看每个关键词的资源数，支持恢复 |

### 3.2 博主关注管理

| 功能 | 描述 |
|------|------|
| 添加关注 | 输入 UP 主名称（Bilibili）或频道名称（YouTube），自动搜索并关注 |
| Bilibili 关注 | 通过 Bilibili 搜索 API 查找 UP 主，支持空间 API 和搜索 API 回退 |
| YouTube 关注 | 通过 YouTube Search API 查找频道 |
| 启用/暂停 | 切换博主的收集状态 |
| 归档博主 | 不再收集该博主内容，已收集的资源保留 |
| 恢复博主 | 将已归档的博主重新激活 |
| 永久删除 | 删除博主及其关联的所有资源（排除收藏资源） |
| 博主总览 | 展示所有博主（含已归档），支持恢复 |

### 3.3 博主内容收集

| 功能 | 描述 |
|------|------|
| 触发收集 | 用户手动触发，系统从所有已启用的博主收集最新视频 |
| 独立于关键词 | 博主收集不经过关键词搜索流程，不使用热度/质量阈值 |
| 时间筛选 | 仅按发布时间收集近期内容（默认近 7 天） |
| 进度反馈 | 实时展示收集进度（轮询 `/api/creators/collect/progress/:id`） |
| 自动定时任务 | 每天早上 08:00 自动执行全量博主内容收集（node-cron） |
| Bilibili 回退 | 空间 API 限流时自动回退到搜索 API，按发布排序筛选该 UP 主视频 |

### 3.4 视频 URL 提交

| 功能 | 描述 |
|------|------|
| 视频链接提交 | 用户粘贴 Bilibili 或 YouTube 视频链接，系统自动提取视频信息 |
| 平台选择 | 支持 Bilibili 和 YouTube 双平台，带有对应平台的 UI 主题色 |
| 信息提取 | 自动获取视频标题、作者、播放量、发布时间等元数据 |
| 热度计算 | 根据播放量（viewCount）换算热度分：`Math.min(100, Math.round(viewCount / 1000))` |
| 去重保护 | 同一 URL 不能重复提交 |

### 3.5 学习资源搜索（关键词）

| 功能 | 描述 |
|------|------|
| 触发搜索 | 用户手动触发，在所有启用的关键词下跨源搜索 |
| 质量阈值 | 关键词搜索的视频需满足最低播放/点赞/收藏阈值 |
| 进度反馈 | 实时展示搜索的阶段（轮询 `/api/monitor/progress/:id`） |
| 结果存储 | 搜索结果持久化到 SQLite 数据库 |
| 综合评分 | 每条资源按公式计算综合分并排序展示 |
| 去重机制 | 跨来源内容去重，采用 Bigram Jaccard 相似度算法 |
| 通知面板 | 搜索完成后弹出通知，展示新增资源条目 |

### 3.6 资源管理

| 功能 | 描述 |
|------|------|
| 资源列表 | 分页展示资源，按热度倒序排列，支持按来源筛选，支持按资源类型切换 |
| 收藏资源 | 点击星标收藏，收藏后移出资源总览，仅在收藏标签页显示 |
| 取消收藏 | 取消收藏后检测关联实体是否存在：实体存在则回到总览，已删除则级联删除 |
| 完成标记 | 标记学习完成，视觉反馈（绿色边框 + 删除线样式） |
| 单条删除 | 含确认对话框防止误操作 |
| 选择式批量删除 | 进入选择模式勾选多个资源，统一删除 |
| 一键清空 | 清空当前分类（关键词/博主/视频）所有资源，含确认弹窗 |
| 知识搜索 | 全文搜索已有资源，搜索结果支持分页浏览 |
| 三种资源分类 | 资源总览按「关键词资源」「博主资源」「视频资源」三个子标签页展示 |

### 5.2 数据库需求

- SQLite 数据库，WAL 模式提高并发读写性能
- 数据库文件位置可通过 `DB_PATH` 环境变量自定义
- 支持渐进式迁移（ALTER TABLE ADD COLUMN + try/catch）

**数据库表结构**：

| 表名 | 说明 |
|------|------|
| `users` | 用户表（id, email, password_hash, role, status, created_at） |
| `verification_codes` | 邮箱验证码表（id, email, code, type, expires_at, used） |
| `keywords` | 关键词表（id, term, scope, enabled, archived, created_at, last_matched_at, user_id） |
| `followed_creators` | 关注博主表（id, platform, channel_id, channel_name, description, avatar_url, enabled, archived, created_at, last_fetched_at, user_id） |
| `news_items` | 学习资源表（id, keyword_id, title, url, source, source_name, published_at, is_real, confidence, summary, matched_at, is_urgent, heat, creator_id, creator_name, completed, favorited, resource_type, user_id） |
| `hot_topics` | 热点话题表（id, title, url, source, source_name, heat, published_at, scope, summary, fetched_at） |
| `config` | 系统配置表（key-value 结构） |
| `user_config` | 用户配置表（key, user_id, value，复合主键） |

### 5.3 性能

- 搜索请求异步执行，前端通过 500ms 间隔轮询进度
- 数据库 WAL 模式提高并发读写
- 博主收集和关键词搜索独立执行，互不阻塞
- ProgressEmitter 在任务完成 5 分钟后自动清理，防止内存泄漏

### 5.4 安全

- 密码通过 bcrypt 哈希存储（10 轮 salt）
- API 密钥通过 AES-256-GCM 加密存储于 `user_config` 表
- API 密钥在响应中自动掩码处理（显示前缀 3 位 + **** + 后缀 4 位）
- JWT 认证中间件保护所有需要登录的 API
- 用户数据隔离：所有查询均包含 `AND user_id = ?` 条件
- adminRequired 中间件保护管理后台 API
- 图形验证码防止机器人批量注册
- 前端通过 Vite proxy 转发 API 请求，避免跨域问题
- 管理员自动分配：仅 SMTP_USER 匹配的邮箱注册时自动获得 admin 角色
- mihomo 配置文件和订阅 URL 存储在服务器本地，订阅 URL 在数据库中加密存储

## 6. 综合评分公式（关键词搜索）

```
综合分 = 热度(heat) × 0.4
       + 来源可信度(credibility) × 0.3
       + 时效性(timeliness) × 0.2
       + 内容类型(contentType) × 0.1
```

| 维度 | 权重 | 取值规则 |
|------|------|----------|
| 热度 | 0.4 | 基于播放量/点赞数等指标归一化 |
| 来源可信度 | 0.3 | 编程导航/鱼皮AI导航=0.9, Bilibili=0.6, YouTube=0.8 |
| 时效性 | 0.2 | 7 天内=1.0, 30 天内=0.7, 90 天内=0.4, 更早=0.2 |
| 内容类型 | 0.1 | AI 课程=0.9, 教程文章=0.8, 视频=0.7 |

> 注：博主内容收集不经过评分系统，仅按发布时间排序。

### 去重机制

跨来源内容去重采用 Bigram Jaccard 相似度算法：
- **标题相似度阈值**：0.75
- **来源相似度阈值**：0.6
- 检测到重复时，保留来源可信度更高的条目

### 来源信用评级

| 来源 | 可信度 | 说明 |
|------|--------|------|
| 编程导航 | 0.9 | 人工筛选的技术文章，质量最高 |
| 鱼皮AI导航 | 0.9 | 人工筛选的 AI 课程，质量最高 |
| YouTube | 0.8 | 国际视频教程，质量高 |
| Bilibili | 0.6 | 视频教程，覆盖面广 |

## 7. API 接口一览

### 7.1 认证

|------|------|------|
| GET | `/api/keywords` | 获取活跃关键词列表 |

## 8. 设计约束

### 8.1 管理后台与主前端分离

- **主前端**：访问 `localhost:5173`，React SPA，面向普通用户
- **管理后台**：访问 `localhost:3001/admin`，独立 HTML/CSS/JS 单页面，面向管理员
- 两个界面完全独立，互不影响，共享同一套后端 API
- 管理后台无需构建工具，由 Express 静态文件服务直接提供

### 8.2 管理员分配规则

- 不引入新的环境变量
- 直接复用 SMTP_USER 配置：注册邮箱与 `SMTP_USER` 匹配时自动设为 `admin` 角色
- 其他邮箱注册均为 `user` 角色

### 8.3 用户配置覆盖规则

- 用户配置的 API 密钥/代理地址优先于任何全局值
- 数据源开关默认开启（用户未设置时视为 `true`）
- 用户配置存储在 `user_config` 表，API 密钥加密存储
- 用户可通过前端设置页清除个人配置，恢复为系统默认行为

### 8.4 代理管理

- 用户输入机场订阅链接后，服务器自动启动 mihomo 实例，无需手动配置代理地址
- 同一订阅链接（相同 URL）的所有用户共享同一个 mihomo 实例，节省服务器资源
- 每个 mihomo 实例占用一个独立端口（15733 ~ 15833），端口自动分配和回收
- mihomo 进程崩溃后自动重启（2 秒延迟），保证服务持续可用
- 代理目标地址（YOUTUBE_PROXY_URL）由系统自动写入用户配置，对用户透明
- 服务器重启后自动扫描数据库恢复所有代理实例
- 代理端口上限 100 个，超出时拒绝新的订阅注册
